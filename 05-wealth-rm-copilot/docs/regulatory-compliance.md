## 1. Regulation Best Interest — 17 CFR 240.15l-1

### Overview

Regulation Best Interest (Reg BI), effective June 30, 2020, requires broker-dealers to act in the best interest of retail customers when making a recommendation of a securities transaction or investment strategy. This agent assists RMs in generating recommendations — the critical architectural decision in this agent is that **suitability determination is Python-only, not LLM output.**

### Reg BI's Four Core Obligations

**1. Care Obligation (17 CFR 240.15l-1(a)(2)(i))**

The RM must have a reasonable basis to believe the recommendation is in the client's best interest based on the retail customer's investment profile.

| Care Obligation Element | Agent Implementation |
|------------------------|---------------------|
| Understand client investment profile | Client profile lookup assembles: risk tolerance, investment horizon, liquidity needs, tax situation, financial goals, existing holdings |
| Understand costs and reasonably available alternatives | Recommendation engine includes cost comparison across alternatives; LLM draft includes cost disclosure |
| Quantitative suitability | Portfolio analysis node calculates concentration, duration, correlation |

**2. Conflict of Interest Obligation (17 CFR 240.15l-1(a)(2)(iii))**

Broker-dealers must establish policies to identify and disclose conflicts of interest.

| Conflict Type | Agent Control |
|--------------|--------------|
| Proprietary product preference | Recommendation engine flags proprietary products and requires explicit RM documentation of why proprietary is in best interest |
| Revenue-sharing arrangements | Prohibited securities list (configured per firm) excludes products with undisclosed revenue sharing |
| Compensation-based incentives | Compensation structure is not an input to the recommendation engine |

**3. Disclosure Obligation (17 CFR 240.15l-1(a)(2)(ii))**

Material facts about the recommendation and conflicts must be disclosed.

The compliance_review node validates that every INVESTMENT_PROPOSAL and REBALANCING_PROPOSAL output includes:
- Explicit statement of recommendation basis
- Costs (fees, loads, spreads) for recommended security
- Comparison to reasonably available alternatives
- Any conflicts of interest (firm relationship with issuer, proprietary product status)
- Form CRS reference

**4. Compliance Obligation (17 CFR 240.15l-1(a)(2)(iv))**

Firms must establish, maintain, and enforce written policies and procedures reasonably designed to achieve compliance with Reg BI.

This agent is part of the firm's Reg BI compliance program:
- Every recommendation generated by this agent is reviewed by the compliance_review node before reaching the RM
- The RM approval gate is a mandatory HITL interrupt — no recommendation reaches the client without RM sign-off
- All approved and rejected recommendations are logged to the audit trail

## 2. Python-Only Suitability Determination — Architectural Rationale

The most consequential design decision in this agent is that the suitability determination — the determination of whether a product is suitable for this client given their investment profile — is made entirely by Python code, not by the LLM.

```python
# From agent/nodes.py — suitability_check_node()
# This function does NOT call Claude or any LLM.
# Suitability is a deterministic compliance function.

def check_suitability(client_profile: ClientProfile, recommendation: Recommendation) -> SuitabilityResult:
    violations = []

    # Risk tolerance check
    if recommendation.risk_rating > client_profile.risk_tolerance_numeric:
        violations.append(SuitabilityViolation(
            type="RISK_MISMATCH",
            detail=f"Product risk {recommendation.risk_rating} exceeds client tolerance {client_profile.risk_tolerance_numeric}",
            severity="HARD_BLOCK"
        ))

    # Liquidity check
    if recommendation.min_holding_period_days > client_profile.liquidity_horizon_days:
        violations.append(SuitabilityViolation(
            type="LIQUIDITY_MISMATCH",
            detail="Client liquidity needs conflict with minimum holding period",
            severity="HARD_BLOCK"
        ))

    # Concentration check
    if recommendation.proposed_weight > CONCENTRATION_LIMITS[recommendation.asset_class]:
        violations.append(SuitabilityViolation(
            type="CONCENTRATION",
            detail=f"Proposed weight {recommendation.proposed_weight}% exceeds {recommendation.asset_class} limit",
            severity="REQUIRES_JUSTIFICATION"
        ))

    # Prohibited securities
    if recommendation.security_id in firm_config.prohibited_securities:
        violations.append(SuitabilityViolation(
            type="PROHIBITED",
            detail="Security is on firm prohibited list",
            severity="HARD_BLOCK"
        ))

    return SuitabilityResult(
        is_suitable=len([v for v in violations if v.severity == "HARD_BLOCK"]) == 0,
        violations=violations
    )
```

**Why this matters for Reg BI compliance:** If the LLM determined suitability, the firm's compliance program would depend on probabilistic AI output for a regulated determination. An LLM could rationalize a risk-mismatched recommendation with plausible-sounding language. The Python code cannot — it enforces hard rules that cannot be reasoned around.

## 3. FINRA Rules — Suitability, Communications, and Books and Records

### FINRA Rule 2111 — Suitability

FINRA 2111 (now interpreted alongside Reg BI for retail customers) requires that recommendations be suitable for the specific customer. The Python suitability check directly implements the three suitability obligations:

| 2111 Obligation | Implementation |
|----------------|----------------|
| Reasonable-basis suitability | Recommendation engine only surfaces securities that pass asset class filters based on firm research |
| Customer-specific suitability | Python suitability check validates against individual client profile |
| Quantitative suitability (churning) | Portfolio analysis node flags excessive proposed trading relative to account size and turnover history |

### FINRA Rule 2210 — Communications with the Public

Client-facing content generated by this agent (meeting summaries, proposal letters, portfolio review narratives) must comply with FINRA 2210's requirement that communications be fair and balanced, not misleading, and include required disclosures.

The compliance_review node validates every client communication against:

| 2210 Requirement | Validation Check |
|-----------------|-----------------|
| Fair and balanced presentation | Checks that risk disclosures are present and not buried |
| No projections of future performance | Scans output for forward-looking performance language; flags for RM review |
| Required disclosure language | Validates Form CRS citation, registration status disclosure |
| Testimonials and endorsements | Not generated by this agent |

### FINRA Rule 4512 — Customer Account Information

Client profile data used by this agent is sourced from the firm's CRM. The agent validates that required 4512 fields are populated before generating investment recommendations:

- Customer name, address, and date of birth
- Investment objectives
- Annual income and net worth
- Risk tolerance
- Investment experience

If any required field is missing, the agent returns a PROFILE_INCOMPLETE status and the recommendation workflow is blocked until the RM updates the CRM.

### FINRA Rule 4511 — Books and Records

All recommendations, proposals, and client communications generated by this agent are logged to the audit trail and stored in S3 with Object Lock GOVERNANCE mode, 6-year retention (FINRA Rule 4511 minimum).

## 4. SEC Investment Advisers Act — 15 U.S.C. § 80b

### Fiduciary Duty (Investment Advisers)

For institutions or personnel registered as investment advisers, the SEC's fiduciary standard (as articulated in the SEC's 2019 guidance) requires loyalty and care. The agent's architecture supports this:

- **Duty of loyalty:** Conflict of interest checks flag proprietary products and revenue-sharing arrangements
- **Duty of care:** Client profile assembly is comprehensive; portfolio analysis includes risk concentration and correlation analysis
- **No material omissions:** Compliance review node validates that client communications include material information

### SEC Rule 204-2 — Investment Adviser Books and Records

Records required under Rule 204-2 that this agent generates and retains:

| Record Type | How Generated | Retention |
|------------|--------------|-----------|
| Recommendation records | Every output logged to DynamoDB + S3 | 5 years |
| Client communications (letters, emails drafted) | Logged with approval status and RM identity | 5 years |
| Suitability analysis records | Python suitability output logged per recommendation | 5 years |
| Trade order records | Not generated by this agent (order entry is separate) | N/A |

## 5. Form CRS — Relationship Summary

### Form CRS Disclosure Requirement

The SEC's Form CRS (Customer Relationship Summary) must be delivered to retail investors. This agent's client-facing outputs reference Form CRS in every document that constitutes a recommendation or advice:

- The compliance_review node validates that Form CRS citation language is present in every INVESTMENT_PROPOSAL and REBALANCING_PROPOSAL output
- The firm's Form CRS URL is a required configuration parameter (set during deployment)
- Meeting prep documents and portfolio review summaries do not require Form CRS citation (they are internal RM tools, not client-facing communications) unless the output is to be shared with the client

## 6. ERISA — 29 U.S.C. § 1001 (Qualified Plan Accounts)

### Fiduciary Duty for Retirement Accounts

For retirement accounts subject to ERISA (401(k), IRA rollovers, pension accounts), this agent applies the ERISA fiduciary standard, which is the highest standard in U.S. securities law.

**ERISA-specific controls in the agent:**

```python
# From agent/nodes.py — suitability_check_node()

if client_profile.account_type in ["IRA", "401K", "PENSION", "RETIREMENT"]:
    # Apply ERISA-specific limits
    erisa_checks = [
        check_prohibited_transaction(recommendation),     # No self-dealing
        check_diversification_requirement(portfolio),     # ERISA § 404 diversification
        check_plan_document_compliance(recommendation, plan_docs)
    ]
    for check in erisa_checks:
        if not check.passed:
            violations.append(SuitabilityViolation(
                type="ERISA_VIOLATION",
                detail=check.detail,
                severity="HARD_BLOCK"
            ))
```

ERISA prohibited transactions (loans to parties in interest, self-dealing, kickbacks) are hard-blocked regardless of any other analysis. The prohibited transaction check runs against the firm's party-in-interest registry.

## 7. 18 U.S.C. § 1960 — No Tipping Off (AML/BSA Separation)

### Architectural Enforcement of No-Tipping-Off

Under 18 U.S.C. § 1956 and 31 U.S.C. § 5318(g), financial institutions and their employees are prohibited from notifying a customer that a SAR has been filed or that their transactions are under investigation. This creates an architectural requirement: the Wealth RM Copilot must not surface AML investigation data to the RM.

**Implementation:**

The Agent 05 application is deployed to a separate Okta application with its own SAML assertion. AML roles (GRP-BSA-Officers, GRP-AML-Analysts) are explicitly NOT assigned to the Agent 05 Okta app:

- An RM logging into Agent 05 cannot see AML investigation status, SAR flags, or Agent 01/02 outputs
- The client profile that Agent 05 receives from Agent 03 (KYC/CDD) contains only: risk tier, document status, and review completion date — no SAR-related fields
- Agent 03's KYC API gateway strips all SAR-related fields before responding to Agent 05 requests

```python
# From 03-kyc-cdd-perpetual-agent/src/api/client_profile.py
# Fields returned to Agent 05 requests (role=RM)

RM_SAFE_FIELDS = [
    "customer_id",
    "risk_tier",           # LOW / MEDIUM / HIGH — no SAR context
    "kyc_status",          # CURRENT / PENDING_REVIEW / EXPIRED
    "last_review_date",
    "document_status"
]

# SAR-related fields — never returned to Agent 05
SAR_RESTRICTED_FIELDS = [
    "sar_flag",
    "sar_investigation_status",
    "agent01_case_id",
    "bsa_review_notes"
]
```

This architectural control is documented for FINRA examination purposes. The no-tipping-off requirement is enforced by design, not by policy.

## 8. SR 11-7 — Model Risk Management

This agent uses an LLM (Claude Sonnet) to draft recommendations, meeting preparation materials, and client communications. SR 11-7 applies to models that inform consequential business decisions.

### Boundaries of LLM vs. Python Roles

| Decision Type | Who Decides | Why |
|-------------|------------|-----|
| Is this recommendation suitable? | Python (`suitability_check_node`) | Deterministic; examiner-auditable; LLM cannot rationalize around hard rules |
| Does this communication meet Reg BI? | Python (`compliance_review_node`) | Compliance checks must be reliable; LLM output is probabilistic |
| What content should the proposal include? | LLM (Claude Sonnet) | Content drafting is where LLM adds value; the compliance gate follows |
| Should the RM approve this recommendation? | Human (RM, HITL interrupt) | Human remains accountable for the recommendation |

**The LLM is the drafting engine. Python is the compliance engine. Humans are accountable.**

### SR 11-7 Documentation

| SR 11-7 Element | Where Documented |
|----------------|-----------------|
| Model purpose and scope | This document + README.md |
| Suitability methodology | `agent/nodes.py` (suitability_check_node) — Python, not LLM |
| Data inputs | `agent/state.py` (WealthCopilotState TypedDict) |
| Validation approach | `tests/` pytest suite |
| Human oversight controls | RM approval gate (mandatory HITL) + compliance_review node |
| Ongoing monitoring | CloudWatch: proposal approval rate, suitability violation frequency |

## 10. Examination Preparedness

### What Examiners Will Ask About This Agent

**"How do you ensure your AI is not making unsuitable recommendations?"**
- Suitability determination is Python code — not LLM output
- Hard blocks on risk mismatch, liquidity mismatch, and prohibited securities cannot be overridden by the LLM or the RM without explicit compliance escalation
- Suitability check output is logged for every recommendation — examiners can audit any decision

**"How does this system comply with Reg BI's Care Obligation?"**
- Client profile is assembled from 12 fields including risk tolerance, liquidity horizon, investment objectives, and financial situation
- Recommendation engine compares proposed securities against client profile before drafting
- Compliance review node validates that required disclosures (costs, conflicts, Form CRS) are present in every client-facing output

**"What happens if the RM doesn't like the AI's recommendation and overrides it?"**
- The RM is the decision-maker — the agent provides a recommendation, not an instruction
- The RM approval gate is a HITL interrupt; the RM can approve, modify, or reject any AI-generated content
- All RM decisions (approve/reject/modify) are logged with the RM's identity and timestamp
- If the RM overrides a suitability violation flag, the override is flagged for compliance officer review

**"How do you prevent the RM from seeing AML investigation data on their client?"**
- Agent 05 is deployed as a separate Okta application with no AML role mappings
- Client profile data returned to Agent 05 is filtered at the API gateway — SAR fields are stripped
- This is architectural enforcement, not policy — the RM cannot see what the system does not return

**"How are you managing model risk under SR 11-7?"**
- LLM role is limited to content drafting; compliance determinations are Python
- Every recommendation goes through a compliance_review node before reaching the RM
- RM approval is mandatory for all client-facing outputs — human accountability is preserved
- Quarterly review of approval rates and suitability violation frequency recommended
