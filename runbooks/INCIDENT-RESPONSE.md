# Runbook: Incident Response

**Scope:** production deployments of any FSI agent (reference architecture: `infra/terraform`).
**Owners:** Platform on-call (primary) · BSA Officer / CCO delegate (compliance incidents) · CISO delegate (security incidents).
**Companion docs:** `MODEL-DEGRADATION-RESPONSE.md` (gradual quality decline — not an incident), `DR-RUNBOOK.md` (infrastructure loss), `offerings/TPRM-DUE-DILIGENCE-PACKET.md` §2 (threat model the classes below map to).

---

## 1. Incident classes and severity

| Class | Examples | Severity | Page |
|---|---|---|---|
| **C1 — Control failure** | OFAC override didn't fire on a live payment; HITL gate bypassed; decision executed without required review | SEV-1 always | Compliance officer + platform on-call, immediately, 24/7 |
| **C2 — AI output incident** | Hallucinated fact discovered in a filed/sent artifact (SAR narrative, adverse-action letter); injection observed steering narrative content | SEV-1 if artifact left the institution; SEV-2 if caught at HITL | Compliance + on-call |
| **C3 — PII exposure** | Unmasked PII found in state, logs, traces, or LLM-bound payloads | SEV-1 if crossed a trust boundary (left VPC / reached external API); SEV-2 in-account | CISO delegate + on-call |
| **C4 — Identity/authz** | HITL approval with unverified identity; `AUTH_DISABLED=true` found in a non-demo environment; role-check bypass | SEV-1 | CISO delegate |
| **C5 — Audit integrity** | Write-ahead audit failures; gaps between state transitions and audit entries; DynamoDB condition-check anomalies | SEV-2 (SEV-1 if concurrent with C1/C2) | Compliance + on-call |
| **C6 — Availability** | Agent down, Bedrock endpoint unreachable, Aurora failover | SEV-2/3 per SLO table in §5 | On-call |

**The one-line triage rule:** if the incident touches a *decision or artifact that has regulatory standing* (a payment released, a SAR filed, an adverse-action letter sent, a collection contact made), it is SEV-1 and compliance is in the room from minute zero.

## 2. First 15 minutes (any class)

1. **Stabilize, don't destroy evidence.** Scale the affected agent service to zero (`aws ecs update-service --desired-count 0`) rather than redeploying. The Aurora checkpointer preserves all in-flight graph state, including paused HITL cases — nothing is lost by stopping; everything is lost by restarting over the evidence.
2. **Freeze the audit picture.** Trigger an immediate WORM snapshot for the affected case(s): `AuditSink.snapshot()` path, or for a stopped service, export the DynamoDB entries (`entry_id` begins with the agent/case prefix) to the Object Lock bucket. COMPLIANCE mode means nobody — including you — can alter it afterward; that is the point.
3. **Establish the blast radius window.** From the audit table, bound the first-bad and last-good `recorded_at` timestamps. Every disposition inside the window is suspect until re-reviewed.
4. **Open the incident record** (ticket + timeline doc). The post-incident review and, for C1/C2, the regulatory-notification decision both depend on a minute-level timeline.

## 3. Class-specific procedures

### C1 — Control failure (worked example: OFAC override failed to fire)
1. Stop the agent (step 2.1). **Manual queue fallback:** route the live alert/payment stream to the pre-agent manual process — every institution retains one as a deployment precondition (pilot SOW requirement).
2. Pull the case's full audit trail; identify which node produced the wrong outcome and with which inputs (`screening_data_missing`, party fields, BIC/IBAN values are all in the entries).
3. **Re-disposition every case in the blast-radius window manually** before anything else. For payments: coordinate holds/recalls with payment ops while OFAC-relevant items are re-screened.
4. Reproduce in a sandbox against the same inputs; write the failing test FIRST (the suite's standing rule: an untested control is a missing control). Fix; the test gates the redeploy through CI.
5. Compliance decides notification obligations (e.g., OFAC voluntary self-disclosure if a blocked-party payment moved; FinCEN contact for SAR-impacting failures). Engineering supplies the timeline; compliance owns the judgment.

### C2 — AI output incident (hallucinated content in a filed artifact)
1. Run `governance.grounding.verify_grounding` against the artifact + its case state; attach the report to the incident. (If grounding passes but the fact is still wrong, the defect is in *state*, i.e., upstream data — different fix, same severity.)
2. If the artifact left the institution: compliance initiates correction per channel (amended SAR per FinCEN instructions; corrected adverse-action notice per Reg B).
3. Sweep: run grounding over every artifact generated by the same prompt version (the `prompt_manifest.json` history bounds the version window precisely).
4. Add the failure as a golden eval case in `governance/evals/golden/` — the regression becomes permanently impossible to reintroduce silently.

### C3 — PII exposure
1. Identify the boundary crossed. In-account (logs/state): purge log streams after preserving samples to the incident record; rotate any implicated secrets. Cross-boundary (external API in a misconfigured env): treat as a reportable privacy event — CISO delegate engages privacy counsel; GLBA/state-breach analysis is theirs, evidence is yours.
2. Find the masking gap: which boundary missed (`platform_core/fsi_agent_platform/pii.py` vs per-agent masking)? Add the leaking pattern to BOTH the masker and a leak-assertion test before redeploy.
3. Check whether the same text reached the LLM (C2 sweep procedure) and the audit trail (entries are immutable — note the exposure in a supplementary entry rather than attempting redaction; immutability beats cosmetic cleanup for examiners).

### C4 — Identity/authz
1. `AUTH_DISABLED=true` outside local demo is an automatic SEV-1: stop the service, inventory every HITL approval taken while it was set (`reviewer_identity_verified: false` in the audit entries makes this a single query), and have a verified reviewer re-approve each.
2. For JWT/role bypasses: capture the token (claims only, never log raw tokens), verify against Cognito/Okta logs, revoke sessions, then the standard fix-test-redeploy path.

### C5 — Audit integrity
1. Write-ahead failures in STRICT mode halt nodes by design — a flood of them usually means DynamoDB throttling/permissions, not tampering. Check `attribute_not_exists` rejections: a rejection means an entry-id collision, which should be impossible (UUID) — treat a real one as tampering evidence and engage CISO.
2. Gap analysis: replay the LangGraph checkpoint sequence (Aurora) against audit entries; every state transition must have its entry. Gaps = the affected cases get manual re-review (C1 procedure step 3).

## 4. Communications and regulatory posture

- **Internal:** SEV-1 bridge within 30 min: on-call, compliance owner, CISO delegate, business owner of the affected workflow. Status notes hourly to the incident channel.
- **Client-facing (SI-operated deployments):** notify the client's named contacts per the SOW's incident SLA (default: SEV-1 within 2 hours, SEV-2 within 1 business day) — with facts and timeline, never speculation about cause.
- **Regulatory:** engineering NEVER communicates with a regulator directly. Compliance owns the channel; this runbook's job is to hand them an evidence pack: timeline, blast radius, audit exports (WORM object keys), re-disposition log, fix evidence (the failing-then-passing test).

## 5. Availability SLOs (C6 reference)

| Signal | Target | Source |
|---|---|---|
| Agent service availability | 99.5% business hours | ALB health checks / Container Insights |
| HITL queue: oldest unreviewed SEV-relevant case | < 4 business hours | see `HITL-QUEUE-OPERATIONS.md` |
| Bedrock invoke error rate | < 1% over 15 min | CloudWatch (model-invocation metrics) |
| Audit write-ahead failures | 0 tolerated (STRICT) | persistence.py logs / DDB throttle alarms |

## 6. Post-incident (within 5 business days)

Blameless review with compliance in the room. Required outputs: (1) timeline; (2) root cause *in control terms* (which control, why it didn't hold); (3) the test that now pins the fix; (4) golden-case/eval additions where applicable; (5) TPRM packet threat-model row update if the incident revealed a new vector; (6) regulatory-notification disposition recorded even when the answer is "none required."
